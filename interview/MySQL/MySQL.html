                        # 事务与锁

                        • 数据库事务的特性

                        ACID：

                        • Atomicity (原子性)

                          事务是一个不可分割的整体，事务内所有操作要么全做成功，要么全失败。

                        • Consistency (一致性)

                          如果数据库中的数据全部符合现实世界中的约束（all defined rules），我们说这些数据就是一致的，或者说符合一致性的。

                        • Isolation (隔离性)

                          多个并发事务之间相互隔离，不能互相干扰。

                        • Durability (持久性)

                          对应的数据库操作所修改的数据都应该在磁盘上保留下来，不论之后发生了什么事故，本次转换造成的影响都不应该被丢失掉。

                        # 分布式架构

                        1. 什么是主从同步，主从同步的作用

                        MySQL 的数据同步机制可以将数据从一个主数据库同步到另一个或多个从数据库，即 Master-Slaves模式。

                        • 数据容灾、备份，数据冗余保证数据不丢失。

                        • 读写分离，可以动态地调整从服务器的数量，从而提高数据库的吞吐量与性能。

                        • 高可用，主从复制 + 故障切换，实现线上业务不宕机运行。

                        1. 主从同步是怎么进行的

                        MySQL 存在三种线程模型，用于主从之间的数据同步：

                        1. Binlog dump thread

                        当从库与主库建立一个连接时，主库中会创建出一个线程（Binlog dump thread），该线程用于当 binlog日志的内容发生变化时，该线程会通知从库，并将相应的 binlog内容发送给从库。

                        1. Slave I/O thread

                        当主从同步开启的时候，从库上会创建2个线程。第一个就是 I/O线程（Slave I/O thread），该线程连接到主库，主库上的 binlog dump线程会将 binlog的内容发送给该 I/O线程。该 I/O线程接收到 binlog内容后，再将变更的内容写入到本地的 relay log。

                        1. Slave SQL thread

                        当 I/O线程将变更的内容写入到本地的 relay log后，SQL线程（Slave SQL thread）会对 relay log进行读取，并且根据 relay log的内容对从库做相应的操作。

                        需要注意的是，当一个主库有多个从库时，在主库上会为每个从库都创建一个 Binlog dump thread，而每个从库都会创建属于自己的 I/O thread与 SQL thread。

                        # 优化方法

                        1. MySQL有两种排序方式filesort和index，一个通过文件排序，一个通过扫描索引本身完成排序。
                        2. 尽量使用覆盖索引，查询中少使用像”SELECT * "，尽量将查询字段和索引字段相匹配。
                        3. 如果ORDER BY 的条件不在索引列上，就会产生 filesort，效率降低，详见实例5
                        4. ORDER BY满足两种情况会使用Index Sort
                          • ORDER BY语句满足最左前缀原则。
                          • WHERE条件与ORDER BY满足最左前缀原则。
                        5. group by与order by很类似，其实质是先排序后分组，遵照索引创建顺序的最左前缀法则。对于group by的优化如果不需要排序的可以加上order by null禁止排序。注意，where高于having，能写在where中的限定条件就不要去having限定了。
                        6. 关联字段加索引，让MySQL做join操作时候尽量选择NLJ算法。
                        7. 小表驱动大表，写多表连接SQL时，如果明知道哪张表是小表可以用straight_join写法固定连接驱动方式，省去MySQL优化区判断的时间。
